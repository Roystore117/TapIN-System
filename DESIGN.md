# Tap-IN 設計ドキュメント

> 最終更新：2026年4月7日  
> ステータス：設計確定・実装中

---

## ゴール

美容院オーナーが毎月手作業でやっている勤怠管理を楽にする ＋ 労基対策

---

## 基本方針

| 役割 | 担当 |
|---|---|
| データの保管・万が一の台帳 | Notion（人間が触る） |
| 打刻・集計・表示・便利機能 | Tap-IN（APIで連携） |

**鉄則：Tap-INが落ちてもNotionだけで手動運用できる状態を常に維持する**

---

## 現在の課題（Before）

```
公休管理・シフト管理（表）
　↓ ①公休をタイムシートに手動転写
個人タイムシート（DB）
　↓ ③タイムシートからMFへ手動転写
マネーフォワード
```

### 問題点
- ① 公休をタイムシートに転写する手間
- ② 従業員枚にタイムシートを毎月作成する手間
- ③ タイムシートからMFへの手入力
- ④ 実働時間の手計算
- ⑤ 打刻が手入力 → 労基リスク

---

## Notion DB構成（update案③）

### DBの役割分担

| DB名 | 書き込む人 | 読む人 |
|---|---|---|
| 従業員マスタDB | 人間（管理者） | Tap-IN（読むだけ） |
| 勤怠DB（公休・打刻情報） | 人間（公休）／Tap-IN（打刻） | Tap-IN（読むだけ） |
| 月次タイムシートDB（新設） | Tap-IN（月末締めで書き込む） | 人間・Tap-IN |
| 個人タイムシート（既存） | 人間 | 人間 |

### 重要：書き込み権限の分離
- 勤怠DBへの公休登録 → 人間が公休カレンダービューから手入力
- 勤怠DBへの打刻登録 → Tap-INのAPIのみ
- **同じDBだが触る場所を分ける**

### 個人タイムシート（既存）の扱い
- バックアップ用として運用継続してもらう
- Tap-INが落ちたときの手動運用の拠り所
- **廃止しない**

---

## Tap-INの役割（After）

```
Notion勤怠DB（公休・打刻情報）
　↓ Tap-INが読み取る
月次集計（月の全日付に展開）← Tap-INが処理
　↓ 月末締めボタンで
月次タイムシートDB（Notionに書き込む）
　↓
タイムシート機能（閲覧・承認）← Tap-IN管理画面
```

### Tap-INがやること
1. 打刻をNotionに自動登録（実装済み✅）
2. 月次日付の生成（30日分）
3. 公休・打刻データを取得して月次集計
4. 集計結果をNotionの月次タイムシートDBに書き込む
5. タイムシート閲覧・承認画面の提供

### Tap-INがやらないこと
- 公休の登録（Notionで人間がやる）
- 個人タイムシートの管理（Notionで人間がやる）
- 承認フローの完全自動化（最初はNotionで）

---

## 月次タイムシートDB書き込みのタイミング

**「月末締めボタン」方式**

1. 月末にオーナーが管理画面で「今月を締める」ボタンを押す
2. Tap-INが勤怠DBから当月データを取得
3. 月次タイムシートDBに一括書き込み
4. 以降は編集不可・承認待ち状態に

メリット：
- タイミングが人間のコントロール下にある
- 締め前なら何度でも打刻修正できる
- 締めたら確定、という自然な運用

## 開発ロードマップ

### フェーズ1（完成済み✅）
打刻 → 勤怠DBに自動登録

### フェーズ2（次にやること👈ここだけ集中）
管理画面（/admin）に「月次集計」タブを追加する

- Notionの勤怠DBから当月データを取得
- 月の全日付に展開して集計
- 画面に表示（MF転記用）

### フェーズ3（後回しでOK）
- 月末締めボタンの実装
- 月次タイムシートDBへの書き込み
- タイムシート閲覧・承認画面

## アプリ構成

**1つのアプリ、画面で役割を分ける**

| パス | 対象 | デバイス |
|---|---|---|
| / | 打刻TOP | スマホ特化 |
| /apply | 時間外申請 | スマホ |
| /admin | 管理画面（集計・設定） | PC特化 |
| /admin/monthly | 月次集計（追加予定） | PC |

## Notion埋め込み

next.config.jsに追加でNotionページ内にTap-INを埋め込み可能。

```javascript
// 本番前はNotionドメインのみに制限すること
headers: [
  {
    key: 'Content-Security-Policy',
    value: 'frame-ancestors https://www.notion.so https://notion.so',
  }
]
```

## リスクヘッジ

| リスク | 対策 |
|---|---|
| Tap-INが落ちる | Notionに生データが残る。個人タイムシートで手動運用 |
| DBデータが壊れる | Tap-INは勤怠DBに追記のみ。削除・上書きしない |
| Notionとの書き込み競合 | 公休は人間、打刻はTap-INで触る場所を分離 |

## 現在のNotion DB ID（環境変数）

```
TIMELOG_DB_ID          = ①勤怠ログDB（打刻・公休・有給）
EMPLOYEE_DB_ID         = ②従業員マスタDB
STORE_SETTINGS_DB_ID   = ③店舗設定DB（店舗名・定休曜日・給与計算設定を統合）
TIPS_DB_ID             = ④保健師の一言DB
OVERTIME_REQUEST_DB_ID = ⑤時間外申請DB
```
